# PolyStudio 多角色一致性扩展方案

## 一、现有架构分析

### 1.1 当前长视频生成流程

```
分镜生成（等用户确认）
       ↓
按分镜为每个镜头生图（单角色一致性机制）
       ↓
图生视频（每个镜头）
       ↓
拼接成片
```

**核心代码位置**：

- Agent 入口：`backend/app/services/agent_service.py`。
- 工作流 Prompt（92-149行）：`backend/app/services/prompt.py`。
- 生图工具：`backend/app/tools/volcano_image_generation.py`。
- 图生视频：`backend/app/tools/volcano_video_generation.py`。
- 视频拼接：`backend/app/tools/video_concatenation.py`。

### 1.2 现有单角色一致性机制

当前 Prompt（第135-148行）规定的角色一致性规则：

```
角色一致性规则（必须遵守）：
- 如果用户提到角色，但对话历史中没有该角色的图片：
  → 必须提示用户先为该角色生成一张角色图

- 如果对话历史中已有角色图：
  → 有角色的镜头：使用 edit_volcano_image_tool，基于已有角色图进行 edit
  → 空镜/场景镜头：使用 generate_volcano_image_tool 生成场景图
```

**现有机制的局限**：

- 仅支持**一个角色 + 多镜头**的简单场景。
- 所有出现该角色的镜头都基于**同一张角色参考图**做 edit。
- 无法处理**多角色同镜**、**多角色多镜头**的复杂场景。

## 二、问题定义

### 2.1 扩展目标

支持同一部长视频中出现**多个不同角色**（如角色 A、角色 B），每个角色在多镜头中形象一致，且角色之间互不混淆。

### 2.2 需要解决的核心问题

| 问题维度   | 现状              | 扩展目标                   |
| ---------- | ----------------- | -------------------------- |
| 角色数量   | 单一角色          | 多个角色（N个）            |
| 角色关系   | 无交叉            | 支持同镜多角色             |
| 参考图管理 | 单一参考图        | 角色ID → 参考图映射        |
| 镜头描述   | 无角色标识        | 明确标注本镜头包含哪些角色 |
| 生图决策   | 简单判断有/无角色 | 根据角色选择正确的参考图   |

## 三、数据结构设计

### 3.1 角色资产表（Character Asset Registry）

在分镜脚本中引入**角色资产表**，显式声明所有角色及其参考图路径：

```json
{
  "video_title": "森林冒险记",
  "total_duration": 30,
  "characters": [
    {
      "id": "char_A",
      "name": "小兔子",
      "reference_image": "/storage/images/char_rabbit_001.jpg",
      "description": "白色绒毛、红色眼睛、戴着蓝色蝴蝶结"
    },
    {
      "id": "char_B",
      "name": "小狐狸",
      "reference_image": "/storage/images/char_fox_001.jpg",
      "description": "橙色皮毛、尖耳朵、绿色眼眸"
    }
  ],
  "shots": [
    {
      "shot_id": 1,
      "duration": 5,
      "description": "小兔子在森林里蹦蹦跳跳",
      "characters_in_shot": ["char_A"]
    },
    {
      "shot_id": 2,
      "duration": 5,
      "description": "小狐狸从草丛中探出头",
      "characters_in_shot": ["char_B"]
    },
    {
      "shot_id": 3,
      "duration": 5,
      "description": "小兔子和小狐狸相遇，互相打招呼",
      "characters_in_shot": ["char_A", "char_B"]
    }
  ]
}
```

### 3.2 关键字段说明

| 字段                           | 类型     | 说明                                     |
| ------------------------------ | -------- | ---------------------------------------- |
| `characters[].id`              | string   | 角色唯一标识符（如 `char_A`、`char_B`）  |
| `characters[].name`            | string   | 角色名称（如"小兔子"），用于自然语言描述 |
| `characters[].reference_image` | string   | 角色参考图路径，用于 edit 工具调用       |
| `characters[].description`     | string   | 角色关键特征描述，用于 prompt 构建       |
| `shots[].characters_in_shot`   | string[] | 本镜头涉及的角色ID列表                   |

## 四、Prompt 扩展方案

### 4.1 分镜生成 Prompt 扩展

在现有分镜生成逻辑中增加**多角色识别**步骤：

```
【扩展后的分镜生成逻辑】

在生成完整分镜脚本前，必须执行以下子步骤：

1. **角色盘点**
   - 分析用户需求中提到的所有角色
   - 检查对话历史中是否已有这些角色的参考图
   - 如果用户提到新角色但没有参考图，提示用户：
     "您的故事中提到了角色 [X]，请先为该角色生成一张参考图"

2. **角色资产确认**
   - 为每个确认的角色分配唯一角色ID（char_A, char_B, char_C...）
   - 记录每个角色的参考图路径

3. **分镜脚本生成**
   - 每个镜头必须明确标注 `characters_in_shot` 字段
   - 空镜标注为空数组 `[]`
   - 单角色镜头标注如 `["char_A"]`
   - 多角色同镜标注如 `["char_A", "char_B"]`
```

### 4.2 生图决策 Prompt 扩展

根据镜头中的角色列表，选择正确的生图策略：

```
【生图决策规则】

对于分镜脚本中的每一个镜头，执行以下决策：

1. **空镜判断**
   - 如果 `characters_in_shot` 为空数组 `[]`
   - → 使用 `generate_volcano_image_tool` 生成纯场景图

2. **单角色镜头**
   - 如果 `characters_in_shot` 包含单个角色ID如 `["char_A"]`
   - → 使用 `edit_volcano_image_tool`
   - → image_url = char_A 的 reference_image
   - → prompt = 角色特征描述 + 镜头场景描述

3. **多角色同镜**
   - 如果 `characters_in_shot` 包含多个角色ID如 `["char_A", "char_B"]`。
   - → 策略A（优先）：使用 `edit_volcano_image_tool`
     * 选择一个角色（char_A）作为主参考图
     * prompt 中明确包含另一个角色（char_B）的特征描述
     * 依赖模型的跨角色编辑能力
   - → 策略B（备选）：分步生图后合成
     * 分别为每个角色生成独立图片
     * 使用图像合成工具（如 PS API 或其他）将多角色合并到同一场景
   - → **重要**：必须确保角色之间互不混淆

4. **同角色多镜头一致性保证**
   - 同一个角色在不同镜头中使用**相同的 reference_image**
   - edit prompt 中始终包含该角色的关键特征描述
   - 不得使用 generate_volcano_image_tool 重新生成（会丢失一致性）
```

## 五、工具层扩展

### 5.1 现有工具能力评估

| 工具                     | 现有能力                 | 多角色场景下的限制     |
| ------------------------ | ------------------------ | ---------------------- |
| `generate_volcano_image` | 文生图                   | 无法保持角色一致性     |
| `edit_volcano_image`     | 单参考图 + prompt → 新图 | 一次只能使用一张参考图 |
| `generate_volcano_video` | 图生视频                 | 依赖前端传入的图片     |

### 5.2 工具调用模式扩展

**扩展后的 edit_volcano_image 调用模式**：

```python
# 单角色镜头
edit_volcano_image_tool(
    prompt="小兔子在森林里蹦蹦跳跳，阳光透过树叶洒落",
    image_url="/storage/images/char_rabbit_001.jpg",  # char_A 的参考图
    size="16:9"
)

# 多角色镜头（以 char_A 为主参考，char_B 特征融入 prompt）
edit_volcano_image_tool(
    prompt="小兔子和小狐狸在森林里相遇，小狐狸从草丛探出头，橙色皮毛、尖耳朵",
    image_url="/storage/images/char_rabbit_001.jpg",  # char_A 作为主参考
    size="16:9"
)
```

### 5.3 潜在的新工具需求

| 工具名称                     | 功能描述                   | 必要性                                 |
| ---------------------------- | -------------------------- | -------------------------------------- |
| `batch_edit_volcano_image`   | 支持多参考图的批量编辑     | 可选（可循环调用现有工具）             |
| `composite_images`           | 将多张角色图合成到同一场景 | **强烈建议开发**（解决多角色同镜问题） |
| `extract_character_features` | 从角色参考图中提取特征向量 | 未来增强用（可选）                     |

## 六、Agent 工作流扩展

### 6.1 扩展后的完整工作流

```
┌─────────────────────────────────────────────────────────────────┐
│                        1. 需求分析与角色盘点                       │
│   - 分析用户需求中的角色数量                                        │
│   - 检查对话历史中的角色参考图                                      │
│   - 如有新角色，提示用户先生成参考图                                  │
└─────────────────────────────────────────────────────────────────┘
                               ↓
┌─────────────────────────────────────────────────────────────────┐
│                        2. 分镜脚本生成                             │
│   - 生成完整的分镜脚本（含角色资产表）                                │
│   - 每个镜头标注 characters_in_shot                               │
│   - 输出给用户确认                                                │
└─────────────────────────────────────────────────────────────────┘
                               ↓
┌─────────────────────────────────────────────────────────────────┐
│                        3. 角色参考图确认                           │
│   - 验证所有角色的 reference_image 是否有效                         │
│   - 如有缺失，提示用户补充                                         │
└─────────────────────────────────────────────────────────────────┘
                               ↓
┌─────────────────────────────────────────────────────────────────┐
│                        4. 批量生图（按镜头）                        │
│   ┌──────────────────────────────────────────────────────────┐  │
│   │ 对每个镜头执行：                                            │  │
│   │ - 空镜 → generate_volcano_image                           │  │
│   │ - 单角色 → edit_volcano_image（使用该角色参考图）             │  │
│   │ - 多角色 → edit_volcano_image（主角色参考图 + 副角色特征）     │  │
│   │ - 记录每个镜头的生图结果（含使用的参考图）                       │  │
│   └──────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
                               ↓
┌─────────────────────────────────────────────────────────────────┐
│                        5. 图生视频                                │
│   - 使用 generate_volcano_video（mode="image"）                  │
│   - 每个镜头独立生成                                               │
└─────────────────────────────────────────────────────────────────┘
                               ↓
┌─────────────────────────────────────────────────────────────────┐
│                        6. 视频拼接                                │
│   - 使用 concatenate_videos 按分镜顺序拼接                          │
└─────────────────────────────────────────────────────────────────┘
```

### 6.2 关键决策点的 Agent 行为

| 决策点         | 条件                      | Agent 行为                                                 |
| -------------- | ------------------------- | ---------------------------------------------------------- |
| 角色盘点       | 用户提到新角色            | 提示："请为 [角色名] 生成一张参考图"                       |
| 分镜生成       | 生成前                    | 自动插入角色资产表                                         |
| 生图策略       | `characters_in_shot` 为空 | 调用 `generate_volcano_image`                              |
| 生图策略       | 单角色                    | 调用 `edit_volcano_image`（使用该角色参考图）              |
| 生图策略       | 多角色                    | 调用 `edit_volcano_image`（主角色参考图 + 副角色特征描述） |
| 角色一致性检查 | 同一角色在不同镜头        | 验证使用相同的 reference_image                             |

## 八、实现要点总结

### 8.1 最小扩展方案（仅修改 Prompt）

| 修改点                   | 内容                                                                |
| ------------------------ | ------------------------------------------------------------------- |
| `prompt.py` 分镜生成逻辑 | 增加**角色资产表**结构，输出 `characters[]` 和 `characters_in_shot` |
| `prompt.py` 生图决策逻辑 | 扩展决策树，支持多角色同镜的特殊处理                                |
| 代码改动量               | 约 50-80 行 Prompt 文本修改                                         |

### 8.2 推荐扩展方案（Prompt + 工具增强）

| 组件                      | 扩展内容                                         |
| ------------------------- | ------------------------------------------------ |
| Prompt 扩展               | 同上（角色资产表 + 生图决策扩展）                |
| 新工具 `composite_images` | 支持多角色图合成到同一场景（解决多角色同镜问题） |
| 代码改动量                | Prompt 修改 + 1 个新工具（约 200 行）            |

### 8.3 关键风险点

| 风险                   | 描述                                 | 缓解措施                                                         |
| ---------------------- | ------------------------------------ | ---------------------------------------------------------------- |
| **角色混淆**           | 多角色同镜时，模型可能将角色特征混合 | 在 prompt 中明确区分各角色特征描述                               |
| **参考图一致性衰减**   | 多次 edit 后角色形象可能漂移         | 始终使用原始 reference_image，不使用前一次 edit 的结果作为新参考 |
| **同镜多角色编辑效果** | 现有 edit 工具单参考图限制           | 评估是否需要 composite_images 新工具                             |

## 九、结论

本方案通过**引入角色资产表**（`characters[]` + `characters_in_shot`）实现多角色一致性的最小化扩展：

1. **数据结构**：角色 ID → 参考图路径的显式映射。
2. **分镜描述**：每个镜头明确标注包含的角色列表。
3. **生图决策**：根据角色数量选择 `generate` 或 `edit` 策略。
4. **一致性保障**：同一角色始终使用相同的 reference_image。

该方案对现有代码架构的侵入性低，主要通过 Prompt 层面的扩展实现，可作为多角色视频生成能力的基线方案。
